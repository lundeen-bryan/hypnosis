“The Architect” — Interaction Profile & Playbook

Designer-of-systems mindset: methodical, precise, iterative. Use this profile to optimize how you work with ChatGPT.

Archetype: Architect Focus: System design & reusability Mode: Outline → Iterate → Stabilize

Overview

You approach ChatGPT as a systems designer: you prefer layered structure, clear interfaces, and outputs that can be reused (scripts, templates, frameworks). You build toward stable, elegant artifacts rather than one-off answers.

Key idea: Optimize for modularity and versioned refinement rather than novelty.

Why Knowing This Type Helps

  • It clarifies that structure-first prompts fit your cognition best.
  • It highlights your edge: organizing complexity into reusable systems.
  • It reduces friction: you can ask for frameworks and diffs instead of ad hoc outputs.

How to Use the Architect Style for Better Results

  1. Work in layers: “Give me a 5-part outline → expand parts 2 & 4 only.”
  2. Ask for modules: “Produce reusable sections with clear boundaries and labels.”
  3. Iterate by diff: “Refactor v3 for readability; preserve headings and IDs.”
  4. Extract patterns: “List the structural principles and wording templates used here.”
  5. Constrain scope: “Update transitions only; leave content sentences intact.”
  6. Request interfaces: “Define inputs/outputs and a checklist for using this template.”
Tip: Name artifacts consistently (e.g., Architect_Playbook_v1.html, Script_Framework_v3.md) to keep versions traceable.

Prompt Templates (Examples)

  • Outline → Expand: “Create a 6-section framework. Then expand sections 1, 3, and 6 to ~150 words each.”
  • Refactor Without Drift: “Polish for clarity; do not change headings, order, or IDs.”
  • Pattern Extraction: “From this script, list phrasing templates and their intended effects.”
  • Interface Spec: “Document prerequisites, steps, and done-criteria as a checklist.”
  • Diff Request: “Return only modified lines with context (±1 line).”

Interaction Blueprint (One-Page Workflow)

  1. Define the artifact: file type, purpose, audience, constraints.
  2. Request skeleton: headings, IDs, and section purposes.
  3. Lock the interface: inputs/outputs, naming, and acceptance criteria.
  4. Iterate by zone: improve one region at a time (e.g., “Transitions only”).
  5. Stabilize: style pass, accessibility, and print/responsive checks.
  6. Version & archive: save as vN; note changes and open issues.
Deliverables: skeleton → draft → refined → stable Controls: IDs, diffs, checklists

Common Pitfalls & Safeguards

  • Scope creep: Mitigate by specifying what not to change each round.
  • Version drift: Keep a short change log with each iteration.
  • Ambiguous goals: Add done-criteria (e.g., “meets A11Y color contrast”).

Personalization Notes

  • Prefer single-column, responsive documents with a consistent toolbar.
  • Use explicit labels (id= on sections) to enable precise, non-destructive edits.
  • When requesting examples, ask for templates that you can adapt rather than client-facing scripts.
